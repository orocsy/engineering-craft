# New Env Var Checklist

Run this when adding ANY new environment variable, regardless of whether it's
production-required, frontend-public, or local-dev-only. The 4-consumer rule applies
universally — partial wiring is the bug.

## Step 1 — classify the var

Before touching code, decide:

- **Production-required?** (e.g., `JWT_SECRET`, `RESEND_API_KEY`)
  → goes in `env.schema.ts` superRefine block AND deploy.yml AND GitHub Secret
- **Optional with safe default?** (e.g., `LOG_LEVEL=info` default)
  → goes in `env.schema.ts` with `.default('info')` AND optionally deploy.yml
- **Frontend-public?** (`NEXT_PUBLIC_*` prefix)
  → goes in Vercel project env vars + `vercel.json` (NOT in deploy.yml — that's API only)
- **Local dev only?** (e.g., dev tunneling URL)
  → goes in `.env.example` only

## Step 2 — wire all consumers in ONE commit

### Required for ALL vars

- [ ] `apps/api/src/config/env.schema.ts` — add `VAR_NAME: z.string().optional()`
      (or `.min(N)`, `.url()`, `.email()`, etc.)
- [ ] If production-required: add to `superRefine` block

```typescript
.superRefine((data, ctx) => {
  if (data.NODE_ENV === 'production' && !data.VAR_NAME) {
    ctx.addIssue({
      code: z.ZodIssueCode.custom,
      path: ['VAR_NAME'],
      message:
        'Required in production — <one-sentence: what breaks without it>. ' +
        'Set the GitHub secret VAR_NAME (<format hint>) and pass it through ' +
        '.github/workflows/deploy.yml.',
    });
  }
});
```

The error message IS the runbook for the deploy operator. Make it specific.

### Required for backend vars

- [ ] `.github/workflows/deploy.yml` — add `-e VAR_NAME="${{ secrets.VAR_NAME }}"` to
      the docker run line
- [ ] `apps/api/.env.example` — add `VAR_NAME=<placeholder>` (no real value)
- [ ] `docker-compose.yml` — add to the `api` service env list (if locally needed)
- [ ] Test fixtures — `buildValidEnv()`, `buildDisabledBillingEnv()`, `jest-setup.ts`
      should set the var to a test-safe value
- [ ] **GitHub Secret**: verify exists with `gh secret list -R owner/repo | grep VAR_NAME`;
      if missing, set it BEFORE merging the PR

### Required for frontend vars (`NEXT_PUBLIC_*`)

- [ ] `apps/admin/.env.local.example` (or booking) — add the var
- [ ] Vercel project env vars — set in the Vercel dashboard for prod/preview
- [ ] `vercel.json` (if applicable) — exposed/inherited config

### Documentation

- [ ] Runbook / README / docs — describe the var, its format, and where it comes from
- [ ] If runbook recommends a SPECIFIC FORMAT (e.g., `Name <email>` for RESEND_FROM_EMAIL):
      ensure the validator accepts that format. Add tests for both bare and documented
      forms.

## Step 3 — run the parity test

```bash
pnpm --filter api test src/config/env.schema.spec.ts
```

This catches:
- Schema declares production-required → deploy.yml missing the pass-through
- Deploy.yml passes through → schema doesn't declare

If the test fails, fix BEFORE pushing.

## Step 4 — if production-required: verify GitHub Secret exists

```bash
gh secret list -R owner/repo | grep VAR_NAME
```

If missing, you have two options:

**Option A — set it yourself** (for autogenerated values like JWT secrets):
```bash
openssl rand -base64 32 | gh secret set VAR_NAME -R owner/repo
```

**Option B — coordinate with the secret owner** (for external API keys):
- Ask the operator who owns the third-party account to set it
- Block the PR with `needs-secret-set:VAR_NAME` label

NEVER merge a PR that adds a `${{ secrets.X }}` reference if X doesn't exist. The
deploy WILL fail at boot.

## Step 5 — post-merge

After merging:

- [ ] Watch the deploy run: `gh run list --workflow deploy.yml --limit 1`
- [ ] Check container logs after rollout for the boot validation result (no Zod errors)
- [ ] Hit the affected feature manually to confirm it works in prod

## Anti-patterns

- "I'll just add the schema check now and update deploy.yml later" — never happens
- "GitHub Secret is set, that's enough" — secret existence ≠ container env exposure
- "I'll grep for the literal later" — deferred greps never happen
- "The runbook will be updated by docs team next sprint" — operator follows current
  runbook, validator rejects, prod boot fails
