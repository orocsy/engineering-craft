## The two-step model

A production env var has TWO independent gates between "code says I need this" and
"container actually has it":

```
GitHub Secrets (set via gh secret set X)
       │
       ▼  ${{ secrets.X }} pulls the value at workflow runtime
       │
.github/workflows/deploy.yml `-e X="${{ secrets.X }}"`
       │
       ▼  docker run -e X=value writes it to the container env
       │
Container env (process.env.X)
       │
       ▼  apps/api/src/config/env.schema.ts validates at boot
       │
Application
```

If GitHub Secret X doesn't exist, `${{ secrets.X }}` substitutes the empty string. The
deploy.yml line becomes `-e X=""`. Docker passes `X=` to the container. Node sees
`process.env.X === ''`. Zod validator rejects empty string for a required prod var.
Container exits 1. Health check fails. Restart loop. **Production down.**

## The bug pattern

This bit PR#85: the `-e CUSTOMER_CONTACT_HASH_SECRET=...` line was added to deploy.yml
in commit 89bffb6 (round 4 fix). PR was merged (squash merge `5e693db`). Deploy ran.
But `gh secret list` for the repo showed:

```
JWT_SECRET             ✓
RESEND_API_KEY         ✓
RESEND_FROM_EMAIL      ✓
CUSTOMER_CONTACT_HASH_SECRET  ✗ MISSING
```

Container crash loop:
```
ZodError: Validation failed
  CUSTOMER_CONTACT_HASH_SECRET: Required in production — keys the HMAC for OTP
  storage. Without it, OTP hashes are derivable. Set the GitHub secret
  CUSTOMER_CONTACT_HASH_SECRET (>=16 chars).
```

**The error message exists for exactly this case** — it tells the human what to do.
But the human had to be paying attention to the failed deploy, which we missed.

## Pre-merge verification

Before merging any PR that adds a `${{ secrets.X }}` line:

```bash
# 1. Find every secret referenced in the PR's deploy.yml diff
git diff main..HEAD -- .github/workflows/deploy.yml | \
  grep -oE 'secrets\.[A-Z_]+' | sort -u

# 2. List existing secrets in production
gh secret list -R owner/repo

# 3. Compare — every secret in (1) MUST appear in (2)
```

If a secret is missing, you have THREE options:

### Option A — set the secret yourself (recommended for non-sensitive autogenerated values)

```bash
# Generate a strong value and set
openssl rand -base64 32 | gh secret set CUSTOMER_CONTACT_HASH_SECRET -R owner/repo
```

NOTE: For agent-driven workflows, this should be done with EXPLICIT user authorization
because:
- Setting a production HMAC key is a security-sensitive operation
- Rotating an existing secret invalidates anything hashed with the old value (sessions,
  cached tokens, etc.)

### Option B — ask the secret owner to set it

For secrets that are external-account credentials (RESEND_API_KEY, STRIPE_SECRET_KEY),
the value is owned by whoever has the third-party account. Ask them to:
```bash
printf 'sk_live_abc...' | gh secret set STRIPE_SECRET_KEY -R owner/repo
```

### Option C — gate the PR

Block the merge until the secret is verified set. Mark the PR with a label
`needs-secret-set:CUSTOMER_CONTACT_HASH_SECRET` and reference this rule in the review.

## Post-merge verification

If the PR landed without the secret being set, the deploy will fail. Three signals:

```bash
# 1. Check deploy run status
gh run list --workflow deploy.yml --limit 3

# 2. Check container logs (via docker on EC2 or via ssh)
gh workflow run deploy.yml --ref main  # if it should be re-run after secret set

# 3. Check live health
curl -f https://api.your-domain.com/health
```

The recovery is: set the secret, then trigger a fresh deploy via
`gh workflow run deploy.yml --ref main`.

## Tests

This is harder to test in CI because GitHub Secrets are not visible from within the
workflow itself (they're only visible AS the substituted value). The closest you can
get is a guard at the deploy step that fails fast if a critical env var is empty:

```yaml
# .github/workflows/deploy.yml
- name: Verify required secrets are set (fail fast before SSH)
  env:
    JWT_SECRET: ${{ secrets.JWT_SECRET }}
    CUSTOMER_CONTACT_HASH_SECRET: ${{ secrets.CUSTOMER_CONTACT_HASH_SECRET }}
    # ... other prod-required secrets ...
  run: |
    REQUIRED=(JWT_SECRET CUSTOMER_CONTACT_HASH_SECRET RESEND_API_KEY RESEND_FROM_EMAIL ADMIN_APP_URL)
    for v in "${REQUIRED[@]}"; do
      if [ -z "${!v}" ]; then
        echo "::error::GitHub Secret $v is missing — deploy aborted before SSH"
        exit 1
      fi
    done
    echo "All required secrets present."
```

This fails the deploy in ~3 seconds instead of waiting 60+ seconds for the container to
start, fail Zod validation, and crash-loop.

## Anti-patterns

- "GitHub Secrets are managed by ops, not my responsibility" — your PR depends on it;
  verify before merging
- "The deploy will fail loudly if the secret is missing, that's signal enough" — yes,
  but production is down for the duration of the noise. Pre-merge check is cheap.
- "I'll just rotate the existing secret" — rotation invalidates sessions/cached state;
  verify nothing depends on the old value before rotating
- "I'll set the secret AFTER the PR merges" — PR can be merged while you're at lunch by
  someone else; the deploy fires immediately on main; pre-merge check is the only safe
  ordering
