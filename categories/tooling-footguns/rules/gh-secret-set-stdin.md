## The footgun

Common assumption from `--help`:
```
gh secret set NAME --body VALUE      # set to VALUE
gh secret set NAME --body -          # read VALUE from stdin (UNIX convention)
```

Actual behavior:
```
gh secret set NAME --body -          # sets the literal string "-" as the secret value
```

This is NOT a stdin-pipe convention. The `gh` CLI treats `--body` as a literal value
regardless of whether it's `-` or anything else.

## The correct ways to set a secret

```bash
# Method 1: pipe via stdin WITHOUT --body
printf '%s' "$VALUE" | gh secret set NAME

# Method 2: literal value via --body
gh secret set NAME --body 'literal-value-here'

# Method 3: from a file
gh secret set NAME < value.txt
```

For autogenerated secrets (e.g., openssl rand for HMAC keys):

```bash
SECRET=$(openssl rand -base64 32)
printf '%s' "$SECRET" | gh secret set CUSTOMER_CONTACT_HASH_SECRET -R orocsy/luxebook
```

The `printf '%s'` (not `echo`) avoids appending a newline that could break the secret.

## Verification step

After setting, ALWAYS verify:

```bash
gh secret list -R orocsy/luxebook | grep SECRET_NAME
```

This confirms the secret exists. It does NOT show the value (correct — secrets are
write-only via API). To verify the value, redeploy and check that the env-schema
validator passes.

## Tests / safeguards

The env-schema validator at `apps/api/src/config/env.schema.ts` typically requires
`min(16)` length for HMAC secrets. A literal "-" value is 1 char, fails validation,
container crashes. So the bug surfaces at next deploy, not at `gh secret set` time.

To catch it earlier:
- Verify length with `gh secret list` and the timestamp (re-run if the value looks
  suspicious)
- Add a CI step that boots the container against the production secrets in a sandboxed
  workflow before promoting (heavy, but catches drift)

## Anti-patterns

- "I'll use `--body -` because UNIX convention" — wrong convention; gh doesn't
  implement it
- "I'll use `echo`, that pipes" — `echo` adds a newline; use `printf '%s'` for exact
  byte count
- "I'll set then forget; deploy will catch it" — deploy fails noisily but production
  is down for the duration
- "I'll set it, and rely on the validator's error message" — validator errors are clear
  but require seeing the deploy logs at 3am
